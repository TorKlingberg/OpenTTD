# Modular Airport Bug Fixes - 2026-01-25

## Summary
Fixed critical bugs causing aircraft to land in wrong spot and get stuck after landing on modular airports.

## Bugs Identified and Fixed

### Bug #1: AircraftController Called for Modular Airports (CRITICAL)
**Location:** `src/aircraft_cmd.cpp:2317-2348` (AirportGoToNextPosition)

**Problem:**
- `AircraftController(v)` was being called for ALL aircraft, including those on modular airports
- `AircraftController` uses the FTA (Finite State Automaton) system with movement data from preset airports
- For modular airports, this FTA data doesn't exist or is invalid
- This caused aircraft to:
  - Stop mid-air during landing approach
  - Get stuck after landing when trying to move to terminals
  - Use incorrect positions and movement paths

**Root Cause:**
The check for modular airports happened AFTER `AircraftController` returned, which was too late.

**Fix:**
Restructured `AirportGoToNextPosition()` to:
1. Check if airport is modular BEFORE calling `AircraftController`
2. For modular airports:
   - Use custom landing logic (AirportMoveModularLanding) for LANDING/ENDLANDING states
   - Use custom takeoff logic (AirportMoveModularTakeoff) for takeoff states
   - Use custom ground pathfinding (AirportMoveModular) when ground_path_goal is set
   - Skip `AircraftController` entirely for ground movement states (TERM1, HANGAR, HELIPAD1)
3. Only call `AircraftController` for modular airports when in air states (FLYING, etc.)

### Bug #2: FTA Position Used for Modular Airports
**Location:** `src/aircraft_cmd.cpp:1763-1809` (AircraftEventHandler_Flying)

**Problem:**
- Even when detecting a modular airport, the code executed FTA logic:
  ```cpp
  v->pos = current->next_position;  // Invalid for modular airports
  st->airport.blocks.Set(apc->layout[v->pos].blocks);
  ```
- This set `v->pos` based on FTA data, which is invalid for modular airports
- Later when `AircraftController` ran, it tried to access invalid FTA positions

**Fix:**
Separated modular airport landing logic from traditional FTA logic:
1. For modular airports:
   - Find nearest runway tile directly
   - Set landing state without using FTA positions
   - Don't set `v->pos` or airport blocks from FTA
2. For traditional airports:
   - Keep existing FTA logic unchanged
   - Ensures backward compatibility

### Bug #3: Missing v->pos Updates for Modular Airports
**Location:** `src/aircraft_cmd.cpp:1809` (end of AircraftEventHandler_Flying)

**Problem:**
- Code unconditionally updated `v->pos` from FTA for all airports, including modular ones
- This could set invalid positions for modular airports

**Fix:**
- Added check to skip `v->pos` update for modular airports
- Only update `v->pos` for traditional FTA-based airports

## Bug #4: Crash When Erasing Modular Airport Tiles

**Location:** `src/airport_gui.cpp:183-196` (CcBuildAirport callback)

**Problem:**
- The `CcBuildAirport` callback is used for both building tiles AND erasing tiles (via CMD_LANDSCAPE_CLEAR)
- When erasing a tile, after the tile is cleared, it's no longer a station tile
- The callback tried to call `IsAirport(t)` on the cleared tile
- `IsAirport()` internally calls `GetStationType()` which asserts the tile is a station
- This caused a crash: `Assertion failed at line 47 of src/station_map.h: IsTileType(t, TileType::Station)`

**Root Cause:**
Missing tile type check before calling `IsAirport()`. The function has a precondition that the tile must be a station, but this wasn't being verified.

**Fix:**
Added `IsTileType(t, TileType::Station)` check before calling `IsAirport(t)`:
```cpp
/* Check if tile is a station before calling IsAirport (which asserts this) */
if (IsTileType(t, TileType::Station) && IsAirport(t)) {
    // ... existing code
}
```

**Impact:**
- Prevents crash when using eraser tool on modular airport tiles
- Safe to call callback for both building and erasing operations
- No functional change when tile is a station

## Bug #5: NOT_REACHED Crash in AirportMove (CRITICAL)

**Location:** `src/aircraft_cmd.cpp:2433` (AirportMove) and `src/aircraft_cmd.cpp:2371-2377` (AirportGoToNextPosition)

**Problem:**
- For modular airports in FLYING state, code was calling `AirportMove()` with FTA system
- Modular airports use a dummy AT_SMALL FTA which doesn't have proper state transitions
- When `AirportMove()` couldn't find a valid path for the aircraft's state, it hit `NOT_REACHED()` at line 2433
- This triggered a fatal error that tried to show a dialog from a background thread
- Result: App crash with `NSWindow should only be instantiated on the main thread!`

**Error Message:**
```
/Users/tor/ttd/OpenTTD/src/openttd.cpp:1257: Backed-up value was not restored!
[Ap] cannot move further on Airport! (pos X state Y) for vehicle Z
NOT_REACHED()
```

**Root Cause:**
Modular airports don't use the FTA (Finite State Automaton) system. Attempting to call `AirportMove()` for modular airports causes it to fail because there's no valid FTA transition data.

**Fix:**
For modular airports in FLYING state:
1. Only use `AircraftController()` for air movement
2. Call the state handler directly (`AircraftEventHandler_Flying`) instead of `AirportMove()`
3. Never call `AirportMove()` which requires FTA data
4. Added fallback: any unexpected states get logged and reset to FLYING to prevent crashes

**Code Changes:**
```cpp
/* For FLYING state on modular airports, just use AircraftController for air movement.
 * Modular airports don't use FTA system, so don't call AirportMove(). */
if (v->state == FLYING) {
    if (!AircraftController(v)) return;
    /* Keep circling or approach for landing via event handler */
    const AirportFTAClass *apc = st->airport.GetFTA();
    _aircraft_state_handlers[v->state](v, apc);
    return;
}

/* For any other unexpected states on modular airports, log and keep circling */
Debug(misc, 1, "[ModAp] Vehicle {} in unexpected state {} on modular airport {}", v->index, v->state, st->index);
v->state = FLYING;
```

**Impact:**
- Prevents NOT_REACHED crash when aircraft are circling modular airports
- Allows aircraft to properly transition from FLYING to LANDING states
- Provides graceful fallback for unexpected states

## Bug #6: Incorrect Takeoff Runway Selection and Jump into Air

**Location:** `src/aircraft_cmd.cpp:2234-2305` (AirportMoveModularTakeoff) and `src/aircraft_cmd.cpp:2370-2420` (FindModularRunwayTileForTakeoff)

**Problem:**
1. Aircraft was taxiing to the wrong end of the runway (top-right instead of bottom-left)
2. Aircraft was jumping into the air instead of climbing smoothly like stock airports

**Root Cause:**
1. `FindModularRunwayTileForTakeoff` was using complex logic that could select the far end of the runway
2. `AirportMoveModularTakeoff` was using simple z++ increments after a delay, not matching stock airport climb behavior

**Fix:**

**Runway Selection:**
Changed to find the runway end nearest to the aircraft's current position (stand/terminal):
```cpp
static TileIndex FindModularRunwayTileForTakeoff(const Station *st, const Aircraft *v)
{
    /* Find the runway end closest to the current aircraft position */
    for (const ModularAirportTileData &data : *st->airport.modular_tile_data) {
        if (!IsModularRunwayPiece(data.piece_type)) continue;

        bool is_end = (data.piece_type == APT_RUNWAY_END || ...);
        int dist = abs(cx - v->x_pos) + abs(cy - v->y_pos);

        /* Strongly prefer runway ends */
        if (!is_end) dist += 10000;

        if (dist < best_dist) {
            best_dist = dist;
            best_tile = data.tile;
        }
    }
    return best_tile;
}
```

**Smooth Takeoff:**
Implemented gradual climb matching stock airports:
```cpp
static bool AirportMoveModularTakeoff(Aircraft *v, const Station *st)
{
    /* Move forward along runway */
    GetNewVehiclePosResult gp = GetNewVehiclePos(v);

    /* Calculate altitude - gradual climb starting after initial acceleration */
    int z = v->z_pos;
    int target_z = GetAircraftFlightLevel(v, true);

    /* Start climbing after 1 tile of acceleration, climb at ~1.5 pixels per tile */
    if (v->modular_takeoff_progress > TILE_SIZE) {
        int climb_progress = v->modular_takeoff_progress - TILE_SIZE;
        int desired_altitude = GetTileMaxPixelZ(v->modular_takeoff_tile) + 1 + (climb_progress * 3 / 2);

        if (z < std::min(desired_altitude, target_z)) {
            z = std::min(desired_altitude, target_z);
        }
    }

    /* Use SetAircraftPosition for proper viewport and shadow updates */
    SetAircraftPosition(v, gp.x, gp.y, z);

    v->modular_takeoff_progress++;

    /* Lift off when we've climbed high enough or traveled far enough */
    if (v->z_pos >= target_z || v->modular_takeoff_progress > TILE_SIZE * 8) {
        v->state = FLYING;
        AircraftNextAirportPos_and_Order(v);
        return true;
    }
}
```

**Impact:**
- Aircraft now taxi to the nearest runway end (same end used for landing approach)
- Takeoff is smooth with gradual climb like stock airports
- Uses SetAircraftPosition for proper visual updates (viewport, shadow, rotor)

## Bug #7: Landing Rollout and Takeoff Runway End Selection

**Location:** `src/aircraft_cmd.cpp:2218-2241` (AirportMoveModularLanding) and `src/aircraft_cmd.cpp:2370-2406` (FindModularRunwayTileForTakeoff)

**Problem:**
1. After landing, aircraft immediately turned off the runway toward terminal, not realistic
2. Takeoff should always use the same end as landing (bottom-left/SE end) for consistency

**Fix:**

**Landing Rollout:**
Added a rollout phase where aircraft continues along the runway after touchdown:
1. After touchdown, find the opposite end of the runway (at least 2 tiles away)
2. Set ground_path_goal to the rollout point with MGT_ROLLOUT target
3. Aircraft taxis along runway to rollout point
4. When rollout complete, find terminal and taxi there
5. Added `FindModularRunwayRolloutPoint()` helper function

```cpp

if (rollout_point != INVALID_TILE) {

    v->ground_path_goal = rollout_point;
    v->modular_ground_target = MGT_ROLLOUT;
    v->state = TERM1;
} else {
    /* No rollout, go directly to terminal */
    AircraftEventHandler_EndLanding(v, st->airport.GetFTA());
}
```

Added MGT_ROLLOUT constant and handler in `HandleModularGroundArrival()` that transitions to finding a terminal.

**Takeoff Runway Selection and Direction:**

## Bug #9: Spinning After Takeoff

**Location:** `src/aircraft_cmd.cpp:2331`

**Problem:**
After completing takeoff, aircraft would spin around in the air before flying off normally. This was because `AircraftNextAirportPos_and_Order()` was being called, which sets the FTA entry point and causes the aircraft to turn back toward the airport.

**Fix:**
Removed the call to `AircraftNextAirportPos_and_Order()` after modular airport takeoffs:
```cpp
/* For modular airports, don't call AircraftNextAirportPos_and_Order()
 * as it would set FTA entry point and cause the plane to turn around.
 * Just let it continue flying in current direction. */
```

**Impact:**
Aircraft now fly straight after takeoff without spinning around.

## Bug #10: Taxiing Across Grass

**Location:** `src/airport_ground_pathfinder.cpp:111-125`

**Problem:**
Aircraft could taxi across grass tiles within the airport bounds because the pathfinder only checked if tiles belonged to the airport bounding box, not if they were actual airport station tiles.

**Fix:**
Added check to ensure tiles are actual airport tiles:
```cpp
for (int i = 0; i < 4; i++) {
    TileIndex neighbor = tile + TileDiffXY(dx[i], dy[i]);
    if (!IsValidTile(neighbor)) continue;
    if (!st->TileBelongsToAirport(neighbor)) continue;

    /* Must be an actual airport station tile, not just grass within airport bounds */
    Tile t(neighbor);
    if (!IsAirport(t)) continue;

    if (CanTilesConnect(st, tile, neighbor, v)) {
        neighbors.push_back(neighbor);
    }
}
```

**Impact:**
Aircraft only taxi on actual airport tiles, not across grass.
